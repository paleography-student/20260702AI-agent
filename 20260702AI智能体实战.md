## 什么是AI AGENT

**AI Agent（人工智能智能体）** 是一种能够**自主感知环境、做出决策并执行行动**的 AI 系统。与传统对话模型"你问一句、我答一句"的被动模式不同，Agent 具备以下核心能力：

| 核心能力               | 说明                              | 类比          |
| ------------------ | ------------------------------- | ----------- |
| **推理（Reasoning）**  | 将复杂任务拆解为可执行的步骤，进行逻辑推演和因果分析      | 大脑的"思考"过程   |
| **规划（Planning）**   | 制定多步行动计划，动态调整策略，处理子任务的依赖关系      | 项目管理中的甘特图   |
| **记忆（Memory）**     | 短期记忆（当前会话上下文）+ 长期记忆（跨会话的知识积累）   | 人脑的海马体 + 皮层 |
| **工具使用（Tool Use）** | 调用外部 API、搜索引擎、代码解释器、文件系统等扩展能力边界 | 人类使用计算器、地图  |

### 与传统 Chatbot 的区别

| 维度 | 传统 Chatbot | AI Agent |
|------|-------------|----------|
| **交互模式** | 一问一答，被动响应 | 自主规划，主动执行 |
| **任务复杂度** | 单轮简单问答 | 多步复杂任务链 |
| **外部能力** | 仅靠模型内部知识 | 调用工具、搜索、执行代码 |
| **记忆** | 会话内短暂记忆 | 长期记忆 + 知识积累 |
| **典型代表** | ChatGPT 早期版本 | Claude Code、Codex、OpenClaw |

> **一句话总结**：如果说传统 LLM 是一个"博学的顾问"，那么 AI Agent 就是一个"能动手的助手"——它不仅知道答案，还能主动去查资料、写代码、调 API，直到把任务彻底完成。

## 相关概念解释

### 1.token（词元）

token是大模型的最小计算单元，我们输入对话之后，模型通过分词，切分成token序列，根据输入的token，计算每个token的向量值。输出时根据已有的token预测下一个token。

api token：令牌，相当于访问各种服务的密码。



### 2. harness

AI工程方法论的四次变迁：

	1. 提示词工程：你是一个熟练的……专家
	2. 上下文工程：把资料、知识库、联网搜索的网页、上下文记忆都喂给AI。

| Hermes      | 自进化研究框架      | 完整通用 Agent | 主动常驻    | MIT 开源，无锁定      | 长期记忆、自迭代、轻量化自托管，编码为插件      |
| OpenCode    | 纯编码 CLI 执行器  | 编码子组件      | 被动单次会话  | 开源多模型           | 极简、只处理代码，作为上层框架的底层执行单元     |

**智能体产品之间也在互相学习和迭代，不需要纠结它们的优缺点，可以选一个先上手**

### 3. mcp

mcp(model context protocol)，模型上下文协议。由 Claude 开发厂商 Anthropic 在 2024 年 11 月开源推出，目前已经成为 AI Agent、本地知识库、大模型调用外部资源的新一代通用工业标准。由host（模型主体）、client（客户端）、sercer（服务端）构成。市面上的mcp有远程和本地两种，提供一个接口，根据这个协议，唤起mcp服务，获取里面的工具。过去的function calling，每一种工具都由厂商自己研发，现在谁都可以开发符合mcp协议的服务，给大模型接入。

启动AI agent时，他会启动所有接入的mcp服务，如果是远程的，就访问远程服务器并建立连接和初始化，如果是本地服务，也是和本地的client建立通信。

一个mcp里可能包含几十上百个工具，模型获取到工具，规划工具，然后注入上下文。

过多的mcp接入会大幅提高token消耗


### 4. skill

把一套固定的流程、约束写成规范。skill里包含技能自身的描述，标签等元数据、一个封装的业务（可以是prompt、一套工作流、包含一定的脚本、一些案例、关于输出内容的结构等等）。

过多的重复的skill，或者描述太过宽泛的skill同样会浪费token

### agent系统的设计

一个 Agent 系统的设计涵盖多个关键维度：用户意图分析、目标（Goal）、规划（Plan）、上下文构建（Context Builder）、函数调用（Function Calling）、技能与 MCP 路由（Skill & MCP Routing）、记忆（Memory）、会话管理（Session Management）、自我评估（Self-Evaluation）、边界设定与约束（Boundaries & Constraints）、验收与修复流程（Verification & Repair），以及多线程并行管理（Multi-threaded Parallelism）等。

设计得当的 Agent，能够高效、准确地执行任务；设计欠佳的 Agent，则容易在多轮会话中出现以下问题：上下文管理失控、工具注入过多导致干扰、感知与路由能力不足而在局部环节反复纠缠——最终造成上下文膨胀，严重拖累模型的推理能力和表现。

claude code + claude opus4.8 > claude code +deepseekv4 pro >trae + deepseek v4 pro

但是 mcp、skill的恰当使用，以及自身的工程规划能力，可以一定程度地弥补这些差距。

## 大模型的使用

### 本地部署

#### 核心概念：参数、精度与显存的关系

本地部署大模型本质上是在你自己的硬件上运行模型推理。决定能否跑起来的关键因素是 **显存（VRAM）**，它由三个变量共同决定：

##### 1. 模型参数量（Parameters）

模型的"大脑容量"，参数越多，模型通常越聪明，但也越吃资源：

| 量级 | 典型模型 | 参数量 | 定位 |
|------|---------|--------|------|
| **小模型** | Qwen2.5-7B、DeepSeek-7B、Llama-3-8B | 7B~8B | 轻量任务、对话、翻译 |
| **中模型** | Qwen2.5-32B、DeepSeek-V3-Lite、Llama-3-70B | 30B~70B | 复杂推理、代码生成 |
| **大模型** | DeepSeek-V3（671B MoE）、Llama-3-405B | 405B~671B | 顶级推理、多模态 |
| **超大模型** | GPT-5、Claude Opus 4 | 万亿级（未公开） | 目前无法本地部署 |

> **MoE（混合专家）模型注意**：如 DeepSeek-V3 总参数 671B，但每次推理只激活约 37B 参数。显存需要装下全部 671B 的权重，但计算量接近 37B 的水平。

##### 2. 模型精度（Precision / Quantization）

精度决定了每个参数用多少 bit 存储，直接影响显存占用和模型质量：

| 精度                    | 每参数占用           | 说明            | 质量损失    |
| --------------------- | --------------- | ------------- | ------- |
| **FP32**（全精度）         | 4 bytes         | 原始训练精度，推理极少使用 | 无损      |
| **FP16 / BF16**（半精度）  | 2 bytes         | 主流推理精度，质量基本无损 | ≈ 0%    |
| **INT8**（8-bit 量化）    | 1 byte          | 轻度量化，小幅质量下降   | 1~3%    |
| **INT4 / GPTQ 4-bit** | 0.5 bytes       | 重度量化，显著降低显存   | 3~8%    |
| **INT2 / 1.5-bit**    | 0.19~0.25 bytes | 极限量化，质量明显下降   | 10~20%+ |

**精度的权衡逻辑**：
```
显存占用 = 参数量 × 每参数字节数 + 推理开销（KV Cache 等）

FP16  → 质量最高，显存最大
INT8  → 性价比最优，质量几乎不变
INT4  → 消费级硬件首选，质量可接受
INT2  → 极端情况，不推荐日常使用
```

##### 3. KV Cache（推理开销）

除了模型权重本身，推理时还需要额外的显存来存储 **KV Cache**（键值缓存）：

- **取决于**：上下文长度（context length）、模型层数、注意力头数
- **大致范围**：模型权重的 10%~30%
- 上下文越长，KV Cache 越大。128K 上下文比 4K 上下文多消耗数倍显存。

#### 本地部署显存测算表

以下为 **推理（Inference）** 所需显存的粗略估算（不含训练），单位 GB。绿色 = 消费级显卡可跑，黄色 = 需多卡或专业卡，红色 = 极难本地部署。

| 模型参数量          | 精度   | 模型权重    | + KV Cache 开销 | **总需显存**     | 消费级显卡方案                              |
| -------------- | ---- | ------- | ------------- | ------------ | ------------------------------------ |
| **7B**         | INT4 | ~3.5 GB | ~0.8 GB       | **≈ 4.5 GB** | 🟢 RTX 3060 (12G) / RTX 4060 (8G)    |
| **7B**         | INT8 | ~7 GB   | ~1.2 GB       | **≈ 8 GB**   | 🟢 RTX 4060 (8G) 勉强 / RTX 4070 (12G) |
| **7B**         | FP16 | ~14 GB  | ~1.5 GB       | **≈ 16 GB**  | 🟡 RTX 4080 (16G) / RTX 5070         |
| **13B~14B**    | INT4 | ~7 GB   | ~1.5 GB       | **≈ 8.5 GB** | 🟢 RTX 4070 (12G)                    |
| **13B~14B**    | INT8 | ~14 GB  | ~2 GB         | **≈ 16 GB**  | 🟡 RTX 4080 (16G)                    |
| **13B~14B**    | FP16 | ~28 GB  | ~3 GB         | **≈ 31 GB**  | 🔴 RTX 5090 (32G) 勉强 / 需多卡           |
| **32B**        | INT4 | ~16 GB  | ~3 GB         | **≈ 19 GB**  | 🟡 RTX 4090 (24G) / RTX 5090 (32G)   |
| **32B**        | INT8 | ~32 GB  | ~4 GB         | **≈ 36 GB**  | 🔴 需 A6000 (48G) 或双 4090             |
| **32B**        | FP16 | ~64 GB  | ~5 GB         | **≈ 70 GB**  | 🔴 需 A100 (80G) 或 4× RTX 4090        |
| **70B~72B**    | INT4 | ~36 GB  | ~6 GB         | **≈ 42 GB**  | 🔴 需 A6000 (48G) / 双 RTX 4090        |
| **70B~72B**    | INT8 | ~72 GB  | ~8 GB         | **≈ 80 GB**  | 🔴 需 A100 80G                        |
| **70B~72B**    | FP16 | ~144 GB | ~10 GB        | **≈ 155 GB** | 🔴 需 2× H100 / 多卡集群                  |
| **405B**       | INT4 | ~203 GB | ~20 GB        | **≈ 225 GB** | 🔴 需 3× H100 80G                     |
| **671B (MoE)** | INT4 | ~336 GB | ~25 GB        | **≈ 360 GB** | 🔴 需 5× H100 80G / 8× A100 80G       |

> **说明**：
> - 以上为推理专用测算。如果同时训练/LoRA 微调，显存需求大约是推理的 3~4 倍（需要存储梯度、优化器状态）。
> - MoE 模型（如 DeepSeek-V3）总参数量大但激活参数少，显存主要花在存储全部专家权重上，推理速度接近同激活量的密集模型。
> - 使用 `llama.cpp` / `Ollama` 的 GGUF 格式可以部分 offload 到 CPU 内存 + 核显，牺牲速度换取更大的模型。

#### 为什么不建议个人本地部署模型

综合以上分析，对于绝大多数个人用户，**不建议本地部署大模型作为主力使用**，原因如下：

##### 1. 💰 硬件成本高，性价比极低

| 方案                              | 一次性投入     | 能跑的模型               | 体验                    |
| ------------------------------- | --------- | ------------------- | --------------------- |
| RTX 4060 (8G)                   | ~¥2,200   | 7B-INT4             | 勉强可用，推理速度慢            |
| RTX 4090 (24G)                  | ~¥13,000  | 32B-INT4            | 中等体验，模型能力有限           |
| RTX 5090 (32G)                  | ~¥16,000+ | 32B-INT8 / 70B-INT4 | 较好体验，仍远不及云端旗舰         |
| 双 RTX 4090                      | ~¥26,000  | 70B-INT4            | 能跑但功耗惊人               |
| Mac Studio M2 Ultra (192G 统一内存) | ~¥40,000  | 70B-FP16            | 速度慢（内存带宽瓶颈），无 CUDA 生态 |

相比之下，API 调用成本极低：DeepSeek-V3 输入 ¥1/百万 token，输出 ¥2/百万 token。一块 RTX 4090 的钱 ≈ **6500 万 token 的 API 调用**，普通用户几年都用不完。

##### 2. 🧠 模型能力差距巨大

- 你能本地跑的（7B~32B INT4）和云端最强模型（Claude Opus 4、GPT-5、DeepSeek-V4）之间存在 **代际差距**，在复杂推理、代码生成、多模态理解上差距悬殊。
- Agent 场景对模型能力要求极高——需要规划、工具调用、自我纠错。本地小模型在这些方面表现远不及云端旗舰。
- 量化（INT4/INT8）会进一步损失模型质量，尤其是在数学推理和代码生成任务上。

##### 3. ⚡ 功耗、散热、噪音

- RTX 4090 满载功耗 ~450W，双卡 ≈ 900W，相当于一台空调的功耗。
- 持续高负载运行产生大量热量和风扇噪音，不适合放在卧室或工位旁。
- 电费成本：按 0.6 元/度算，24×7 跑一张 4090 每月电费约 ¥200。

##### 4. 🔧 运维门槛高

- 需要折腾 CUDA、cuDNN、PyTorch 版本兼容性。
- 模型下载动辄几十 GB，网络不稳定时体验极差。
- 多用户并发、会话管理、负载均衡等生产环境问题完全需要自己解决。
- 驱动更新、依赖冲突、OOM 崩溃排查耗时耗力。

##### 5. 📉 模型迭代快，硬件迅速贬值

- 模型每 3~6 个月迭代一代，你今天买的显卡刚能跑当前最好的本地模型，半年后可能就跑不动新的了。
- 而云端 API 永远能用最新最强的模型，不需要任何硬件升级。

##### 🎯 唯一值得本地部署的场景

| 场景          | 理由                           |
| ----------- | ---------------------------- |
| **数据绝对隐私**  | 涉密数据、医疗记录、法律文件不能出内网          |

> **结论**：对于 95% 以上的个人用户，**API 调用 + 云端 Agent 产品** 是更理性、更经济的选择。把买显卡的钱省下来订阅 Coding Plan 或 API 额度，体验远好于本地折腾。本地部署的"拥有感"是心理价值，实际投入产出比极低。

**缓存（Cache）与撞缓存现象**

#### 什么是缓存？

在 AI 大模型中，**缓存**指的是将模型处理过的计算结果暂存起来，当再次遇到相同或相似的输入时，直接复用已有结果，而不需要重新计算。

**类比**：就像你把一道复杂数学题的解题过程记在草稿纸上，下次遇到同样的题，直接翻草稿纸就行，不用重新算一遍。

在 LLM 调用中，缓存主要发生在两个层面：

| 层面 | 说明 | 举例 |
|------|------|------|
| **Prompt 缓存** | 对相同前缀的 prompt，复用已计算的注意力矩阵和 KV 值 | 系统提示词（system prompt）每次相同，只计算一次 |
| **上下文缓存** | 长对话中，已处理过的历史消息不再重复计算 | Claude 的 prompt caching：标记 `cache_control` 的块被缓存 |

**缓存的好处**：
- ⚡ **降低延迟**：跳过重复计算，响应更快
- 💰 **降低成本**：缓存命中的 token 通常按折扣价计费（如 Claude 缓存写入按原价，读取按 1/10 价格）
- 🔄 **提升吞吐**：服务端负载降低

#### 什么是"撞缓存"现象？

**撞缓存**（Cache Collision / Cache Poisoning）是指在多用户共享的模型服务中，不同用户的请求因为前缀相同而意外命中同一块缓存，导致：

1. **信息泄露风险**：用户 A 的系统提示词被缓存，用户 B 的请求命中了该缓存，可能看到 A 的上下文片段
2. **行为污染**：前一个用户的对话状态残留在缓存中，影响后续用户的模型输出
3. **调试困惑**：同一个 prompt 在不同时间返回不同结果，因为缓存状态不一致

**为什么会出现？**

```
用户A: [系统提示词A] + "帮我写一封邮件"
         ↓ 缓存了 [系统提示词A] 的计算结果
         
用户B: [系统提示词A] + "总结这份文档"  
         ↓ 命中缓存 → 使用用户A时的缓存状态
         ↓ 如果缓存隔离不严 → 可能掺杂用户A的上下文
```

**注意**：主流模型厂商（OpenAI、Anthropic 等）的缓存是按 API Key 隔离的，正常情况下不会跨用户撞缓存。**撞缓存更多发生在**：
- 自建代理/中转服务（如 API2D、CloseAI 等）缓存策略不当
- 本地部署时多用户共用同一实例
- 使用第三方客户端时，客户端层面的缓存 bug

#### 如何避免撞缓存？

1. **使用官方 API 直连**，减少中间层
2. **避免在中转平台长期使用同一 API Key 跑高频对话**
3. **敏感场景每次新建会话**，不依赖长对话缓存
4. **本地部署时做好用户会话隔离**

### sdk工具包

**SDK（Software Development Kit，软件开发工具包）** 是模型厂商提供的一套封装好的代码库，让开发者用编程语言（Python、JavaScript、Java 等）直接调用大模型，而不需要手写底层的 HTTP 请求。

#### SDK 的工作原理

SDK 本质上是对 **REST API** 的一层封装。整个调用链路如下：

```
你的代码
   ↓ 调用 SDK 方法（如 client.chat.completions.create()）
SDK 层
   ↓ 构造 HTTP 请求（拼接 URL、添加认证头、序列化参数）
网络传输
   ↓ HTTPS 请求 → 模型服务器
模型推理
   ↓ GPU/TPU 计算，生成 token
网络返回
   ↓ HTTPS 响应（或 SSE 流式响应）
SDK 层
   ↓ 解析响应、处理流式数据、错误重试
你的代码
   ↓ 拿到结构化的返回结果
```

#### SDK 帮你做了哪些事？

| 功能 | 没有 SDK（手写） | 有 SDK |
|------|-----------------|--------|
| **认证** | 手动在 Header 里加 `Authorization: Bearer sk-xxx` | `client = OpenAI(api_key="sk-xxx")` 一行搞定 |
| **请求构造** | 手写 JSON，注意嵌套结构、字段类型 | 调用函数，IDE 自动补全，类型安全 |
| **流式输出** | 手动解析 SSE（Server-Sent Events）协议 | `stream=True`，`for chunk in stream:` 直接遍历 |
| **错误处理** | 手动判断 HTTP 状态码、解析错误 JSON | SDK 自动抛出对应异常（`RateLimitError`、`AuthenticationError` 等） |
| **重试机制** | 自己写重试逻辑 + 指数退避 | SDK 内置自动重试（如 OpenAI SDK 默认重试 2 次） |
| **Token 计数** | 需要自己调 tokenizer | SDK 提供 `count_tokens()` 等方法 |
| **连接管理** | 手动管理 HTTP 连接池 | SDK 内置连接池复用，减少握手开销 |

#### 主流 SDK 示例

**OpenAI Python SDK（最经典的范式）：**

```python
from openai import OpenAI

# 1. 初始化客户端
client = OpenAI(api_key="sk-xxx", base_url="https://api.openai.com/v1")

# 2. 调用模型（非流式）
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {"role": "system", "content": "你是一个有帮助的助手"},
        {"role": "user", "content": "解释一下SDK是什么"}
    ],
    temperature=0.7,
    max_tokens=1000
)

# 3. 获取结果
print(response.choices[0].message.content)

# 4. 流式调用
stream = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "写一首诗"}],
    stream=True
)
for chunk in stream:
    if chunk.choices[0].delta.content:
        print(chunk.choices[0].delta.content, end="")
```

**Anthropic Python SDK：**

```python
import anthropic

client = anthropic.Anthropic(api_key="sk-ant-xxx")

response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1000,
    system="你是一个有帮助的助手",
    messages=[
        {"role": "user", "content": "解释一下SDK是什么"}
    ]
)
print(response.content[0].text)
```

#### SDK vs API：该用哪个？

| 维度       | 直接调 API             | 用 SDK          |
| -------- | ------------------- | -------------- |
| **上手难度** | 需要理解 HTTP、JSON、认证机制 | 装个包，看文档，几行代码跑通 |
| **灵活性**  | 完全控制请求的每个细节         | 受限于 SDK 封装的程度  |
| **开发效率** | 低，重复造轮子             | 高，开箱即用         |
| **维护成本** | 接口变更需要手动适配          | SDK 升级即可（厂商维护） |
| **适合场景** | 自定义需求极强、SDK 不支持的语言  | 日常开发、快速原型、生产环境 |

> **建议**：绝大多数情况下直接用官方 SDK。只有 SDK 不支持的场景（如使用小众编程语言、需要自定义网络层）才考虑手写 API 调用。国内厂商（豆包、千问、DeepSeek 等）大多兼容 OpenAI SDK 的接口格式，可以无缝切换。

#### 关键注意事项

1. **API Key 安全**：永远不要把 key 硬编码在代码里。用环境变量（`os.environ.get("OPENAI_API_KEY")`）或 `.env` 文件管理。
2. **base_url**：使用中转平台或国产模型时，需要把 `base_url` 指向对应地址。例如硅基流动：`base_url="https://api.siliconflow.cn/v1"`。
3. **流式输出**：对于长文本生成，务必使用 `stream=True`，否则用户要等很久才能看到第一个字。
4. **异常处理**：生产环境务必 `try/except` 捕获 `RateLimitError`（限流）、`APITimeoutError`（超时）等。

## mcp

MCP（Model Context Protocol）服务是 AI Agent 连接外部世界的"感官和手脚"。每个 MCP 服务提供一组**工具（Tools）**，Agent 可以根据任务需求自主选择和调用。以下介绍 Codex 中常用的 6 个 MCP 服务。

### 1. Brave Search（网络搜索）

**定位**：为 AI Agent 提供全面的互联网搜索能力。

Brave Search MCP 是由 Brave 官方维护的 MCP 服务，基于 Brave Search API，让 Agent 能够自主进行网络搜索。它是 MIT 开源的，支持 STDIO 和 HTTP 两种传输模式。

**提供的工具：**

| 工具                   | 功能                                  |
| -------------------- | ----------------------------------- |
| `brave_web_search`   | 通用网页搜索，支持按国家、语言、时效性、安全等级过滤          |
| `brave_local_search` | 本地商户/地点搜索，返回评分、营业时间、AI 描述（需 Pro 套餐） |
| `brave_video_search` | 视频搜索，返回元数据和缩略图                      |
| `brave_image_search` | 图片搜索，返回 URL、尺寸、置信度（最多 200 条）        |
| `brave_news_search`  | 新闻搜索，支持时效性控制和突发新闻标记                 |
| `brave_summarizer`   | AI 摘要，基于搜索结果生成智能总结                  |
| `brave_place_search` | POI 兴趣点搜索，按坐标或地名返回结构化地点数据           |
| `brave_llm_context`  | 为 LLM 和 RAG 管线优化的预提取网页内容            |

**使用方式：**

```json
{
  "command": "npx",
  "args": ["-y", "@brave/brave-search-mcp-server"],
  "env": { "BRAVE_API_KEY": "你的API密钥" }
}
```

> **免费获取 API Key**：在 [Brave Search API 控制台](https://api.search.brave.com/) 注册，免费套餐每月有一定额度。付费套餐（Search 计划）解锁更多功能和更高频率。

**典型场景**：让 Agent 查资料、验证事实、搜索最新文档、对比方案等。

### 2. Context7（实时文档查询）

**定位**：解决 LLM 训练数据过时导致的"幻觉 API"问题——让 Agent 能查到最新的库文档和代码示例。

Context7 是由 Upstash 开发的开源平台（MIT 许可），专门为 AI 编程工具提供**版本匹配的实时文档**。它的核心价值在于：当你让 Agent 用某个库写代码时，Context7 会拉取该库**当前版本**的官方文档，而不是依赖模型训练时记住的旧 API。

**提供的工具：**

| 工具 | 功能 |
|------|------|
| `resolve-library-id` | 根据库名和查询意图，解析出 Context7 中的库标识符（如 `/vercel/next.js`） |
| `query-docs` | 根据库标识符和具体问题，检索版本匹配的文档和代码示例 |

**核心特性：**

- **版本自动匹配**：提到 "Next.js 14" 会自动检索 v14 对应文档，而非最新或旧版
- **斜杠语法直达**：可直接指定 `/supabase/supabase` 跳过匹配步骤
- **一键安装**：`npx ctx7 setup` 自动完成 OAuth 认证、API Key 生成、MCP 配置
- **社区共建**：文档库由社区贡献，覆盖面持续增长

**使用方式：**

```json
{
  "command": "npx",
  "args": ["-y", "@upstash/context7-mcp"],
  "env": { "CONTEXT7_API_KEY": "你的API密钥" }
}
```

> **免费获取 API Key**：在 [Context7 控制台](https://context7.com/dashboard) 注册，免费 Key 有更高速率限制。

**典型场景**：写代码时自动查最新 API、避免使用已废弃的方法、获取准确的代码示例。

### 3. Fetch（网页内容抓取）

**定位**：让 Agent 能够获取任意网页的内容并转换为可读格式。

Fetch MCP 是 MCP 官方参考服务器之一，功能简洁但极其实用——将任何 URL 的内容抓取下来，把 HTML 转换为 Markdown 格式，方便 LLM 阅读和处理。

**提供的工具：**

| 工具 | 功能 |
|------|------|
| `fetch` | 抓取 URL 内容并返回 Markdown 格式文本 |

**参数说明：**

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `url` | 要抓取的网址（必填） | — |
| `max_length` | 返回内容的最大字符数 | 5000 |
| `start_index` | 起始字符位置（支持分段读取长网页） | 0 |
| `raw` | 是否返回原始 HTML（不转 Markdown） | false |

**核心特性：**

- **分块读取**：通过 `start_index` 参数，可以像翻书一样逐段读取长网页
- **Markdown 转换**：自动将 HTML 转为 Markdown，去除广告和排版噪声
- **robots.txt 遵守**：默认遵守网站的爬虫协议（可通过 `--ignore-robots-txt` 关闭）
- **自定义 User-Agent**：可配置请求头标识
- **代理支持**：可通过 `--proxy-url` 配置代理

**安装方式：**

```bash
# 推荐方式（uvx）
uvx mcp-server-fetch

# 或 pip
pip install mcp-server-fetch
```

> ⚠️ **安全提醒**：Fetch 可以访问本地/内网 IP，需注意安全风险。生产环境建议配置网络隔离。

**典型场景**：抓取在线文档、阅读技术文章、获取 API 文档、提取网页中的关键信息。

### 4. GitHub（代码仓库管理）

**定位**：让 Agent 直接操作 GitHub 仓库——搜索代码、管理 Issue/PR、读写文件。

GitHub MCP 是 MCP 官方参考服务器之一，将 GitHub API 封装为一组工具，让 Agent 能够在你的授权范围内自主管理 GitHub 仓库。它是代码 Agent（如 Codex、Claude Code）的核心基础设施。

**提供的工具（部分）：**

| 工具类别 | 典型工具 | 功能 |
|----------|---------|------|
| **仓库管理** | `search_repositories`、`create_repository`、`fork_repository` | 搜索、创建、Fork 仓库 |
| **文件操作** | `get_file_contents`、`create_or_update_file`、`push_files` | 读写文件、批量推送 |
| **Issue 管理** | `create_issue`、`search_issues`、`update_issue`、`add_issue_comment` | Issue 的全生命周期管理 |
| **PR 管理** | `create_pull_request`、`search_pull_requests`、`merge_pull_request` | 创建、搜索、合并 PR |
| **代码搜索** | `search_code`、`search_commits` | 搜索代码内容和提交记录 |
| **分支管理** | `create_branch`、`get_branch`、`delete_branch` | 分支的增删查 |

**核心特性：**

- **细粒度权限控制**：通过 GitHub Personal Access Token 精确控制 Agent 能访问哪些仓库和操作
- **支持 GitHub Enterprise**：可用于私有部署的 GitHub 实例
- **完整的 Git 操作链**：从创建分支、编辑文件到提交 PR，Agent 可完成完整工作流

**使用方式：**

```json

}
```

> **获取 Token**：GitHub → Settings → Developer settings → Personal access tokens → Fine-grained tokens。建议只授予必要的仓库和权限。

**典型场景**：自动修复 Bug 并提交 PR、跨仓库搜索代码、生成 Release Notes、管理 Issue 工作流。

### 5. Node REPL（JavaScript 代码执行）

**定位**：为 Agent 提供 JavaScript/Node.js 的交互式执行环境。

Node REPL MCP 让 Agent 能够直接运行 JavaScript 代码片段并获取执行结果。当 Agent 需要验证代码逻辑、进行数据计算、测试表达式时，不必"凭空想象"输出，而是真正执行代码并观察结果。

**提供的工具：**

| 工具 | 功能 |
|------|------|
| `execute` | 在 Node.js 环境中执行一段 JavaScript 代码，返回输出结果 |

**核心特性：**

- **真实执行环境**：代码在真实的 Node.js 进程中运行，而非模拟
- **即时反馈**：Agent 写代码 → 执行 → 看结果 → 修正，形成闭环
- **支持异步**：可以使用 `async/await` 语法
- **轻量安全**：运行在沙箱化环境中，适合快速验证

**使用方式：**

```json
{
  "command": "npx",
  "args": ["-y", "node-repl-mcp"]
}
```

> 需要本地安装 Node.js 环境（建议 v18+）。

**典型场景**：验证正则表达式、测试算法逻辑、数据格式转换验证、调试 JavaScript 代码片段、执行数学计算。

### 小结：MCP 服务的选用原则

| 原则       | 说明                                                |
| -------- | ------------------------------------------------- |
| **按需接入** | 不是越多越好——每个 MCP 的工具体系都会注入上下文，过多会挤占 token 预算        |
| **同类择优** | 例如搜索类有 Brave Search 和 WebSearch，选一个最适合的即可         |
| **注意权限** | GitHub、Zotero 等服务涉及你的私有数据，务必使用最小权限 Token          |
| **关注延迟** | 远程 MCP 服务（如 Context7、Brave Search）每次调用都有网络延迟，合理使用 |
| **版本更新** | MCP 生态快速迭代，定期检查所用 MCP 是否有新版本或更好的替代品               |

## codex的配置

Codex 的核心配置由四个关键文件组成，它们共同定义了 Codex 的行为、身份认证和会话管理。

### 1. AGENTS.md —— 项目指令与规范文件

**作用**：AGENTS.md 是 Codex 的"项目说明书"。它让你为 Codex 提供**持久化的项目指令和上下文**。每次 Codex 开始工作时，它会自动读取 AGENTS.md，从而让团队在每个任务开始时都能获得一致的规范。

**发现机制（三级优先级链）**：

1. **全局作用域**：Codex 首先检查 Codex 主目录（默认 `~/.codex`）中的 `AGENTS.override.md`，如果不存在则读取 `AGENTS.md`。只使用第一个非空文件。
2. **项目作用域**：从项目根目录（通常是 Git 根目录）向下遍历到当前工作目录。在每个目录中依次检查 `AGENTS.override.md` → `AGENTS.md` → 后备文件名。
3. **合并顺序**：文件从根目录向下拼接，越接近当前目录的文件越靠后，因此可以**覆盖**前面的指令。

**文件格式**：标准 Markdown，通常包含二级标题和要点规则。

**示例**：

```markdown
# AGENTS.md

## 工作约定

- 修改 JavaScript 文件后必须运行 `npm test`
- 安装依赖时优先使用 `pnpm`
- 添加新的生产依赖前需要确认

## 仓库规范

- 提交 PR 前运行 `npm run lint`
- 修改公共工具函数时同步更新 `docs/` 中的文档
```

**自定义后备文件名**：如果项目使用不同的文件名约定，可以在 `config.toml` 中配置：

```toml
project_doc_fallback_filenames = ["TEAM_GUIDE.md", ".agents.md"]
project_doc_max_bytes = 65536  # 默认 32 KiB
```

**CODEX_HOME 配置文件**：通过设置 `CODEX_HOME` 环境变量，可以为不同场景使用不同的指令配置：

```bash
CODEX_HOME=$(pwd)/.codex codex exec "列出当前激活的指令来源"
```

**注意事项**：
- 空文件会被跳过
- 合并后的内容超过 `project_doc_max_bytes`（默认 32 KiB）会被截断
- 没有缓存机制——每次运行或 TUI 会话启动时重新构建指令链
- 排查问题时可用 `codex status` 验证工作区根目录，或启用 TUI 日志 `codex -c log_dir=./.codex-log`

### 3. auth.json —— 身份认证文件

**作用**：`auth.json` 是 Codex 的**身份凭证缓存文件**。它存储访问令牌（access token），让 Codex 能够以你的身份调用 OpenAI API 或 ChatGPT 服务。CLI 和 IDE 扩展共享此文件——在任一端登出，另一端也需要重新登录。

**位置**：`~/.codex/auth.json`（默认），或由 `cli_auth_credentials_store` 配置决定存储方式。

**认证方式**：

| 方式             | 说明                                                           |
| -------------- | ------------------------------------------------------------ |
| **ChatGPT 登录** | 默认方式，打开浏览器完成登录。提供 Plus/Pro/Business/Edu/Enterprise 订阅访问      |
| **API Key 登录** | 从 [OpenAI 平台](https://platform.openai.com/api-keys) 获取，按用量计费 |

**凭证存储位置控制**：

```toml
cli_auth_credentials_store = "keyring"    # file / keyring / auto
```

- **`file`**：存储在 `~/.codex/auth.json`（明文）
- **`keyring`**：存储在操作系统凭证管理器
- **`auto`**：优先使用系统凭证管理器，不可用时回退到文件

**安全警告**：⚠️ **像密码一样对待 `auth.json`**——它包含访问令牌。不要提交到 git、粘贴到工单、或在聊天中分享。

**Token 自动刷新**：ChatGPT 会话的 token 会在使用中自动刷新，活跃会话通常无需重新登录。

**无头/远程机器登录方案**：

1. **设备码认证**（Beta）：运行 `codex login --device-auth`，在浏览器中输入一次性代码
2. **复制 auth.json**：在有浏览器的机器上完成登录，将 `auth.json` 复制到远程机器
   ```bash
   ssh user@remote 'mkdir -p ~/.codex && cat > ~/.codex/auth.json' < ~/.codex/auth.json
   ```
3. **SSH 端口转发**：`ssh -L 1455:localhost:1455 user@remote`，然后运行 `codex login`

**企业管控**：管理员可限制登录方式：
```toml
forced_login_method = "chatgpt"           # 或 "api"
forced_chatgpt_workspace_id = "uuid"      # 限制到特定工作区
```

### 4. Session —— 会话管理

**作用**：Session（会话）是 Codex 的**对话记录和状态管理系统**。它不是单一文件，而是 `~/.codex/sessions/` 目录下的一组文件集合。每个会话保存完整的对话记录（transcript）、计划历史（plan history）和审批状态（approvals），让你可以随时恢复之前的工作。

**会话存储**：

- 位置：`~/.codex/sessions/`
- 每个会话有唯一标识符（UUID，如 `7f9f9a2e-1b3c-4c7a-9b0e-...`）
- 包含完整的对话转录、计划历史和审批记录

**恢复会话的方式**：

| 命令 | 作用 |
|------|------|
| `codex resume` | 启动交互式选择器，浏览最近的会话 |
| `codex resume --last` | 快速恢复当前目录最近一次会话 |
| `codex resume --last --all` | 恢复所有目录中最近一次会话 |
| `codex resume <SESSION_ID>` | 直接恢复指定会话 |
| `codex resume --all` | 显示所有目录的会话（不只是当前目录） |

**非交互式恢复**（自动化/脚本）：
```bash
codex exec resume --last "修复你发现的竞态条件"
codex exec resume 7f9f9a2e-1b3c-4c7a-9b0e-.... "实现这个计划"
```

**会话内操作**：

| 操作 | 快捷键/命令 | 说明 |
|------|------------|------|
| 清屏 | `/clear` | 清除终端并开始新对话 |
| 纯清屏 | `Ctrl+L` | 清屏但不结束对话 |
| 退出 | `Ctrl+C` 或 `/exit` | 关闭交互式会话 |
| 搜索历史 | `Ctrl+R` | 搜索提示词历史 |
| 浏览草稿 | `↑` / `↓` | 恢复之前的草稿文本和图片 |
| 编辑上条消息 | `Esc Esc`（空输入框） | 编辑上一条用户消息，重复按可继续回溯 |
| 分支对话 | 编辑后按 `Enter` | 从编辑点分叉新对话 |

**会话历史配置**：
```toml
[history]
persistence = "save-all"    # 保存所有会话 / "none" 不保存
max_bytes = 104857600       # 限制历史文件大小，超出后删除最早记录
```

**会话上下文保持**：每次恢复的会话保留原始转录、计划历史和审批记录，Codex 可以**利用之前的上下文**同时接受新指令。可以通过 `--cd` 覆盖工作目录或 `--add-dir` 添加额外根目录。

**会话可见性**：在 TUI 中，你可以实时观察 Codex 在做出更改前解释计划，并逐步骤批准或拒绝。TUI 始终显示 Codex 操作的转录，便于通过标准 git 工作流进行审查和回滚。

### vibe coding

**Vibe Coding** 是由前 OpenAI、特斯拉 AI 负责人 **Andrej Karpathy** 在 2025 年初提出的一个编程新范式。核心理念是：**"最热门的编程语言是英语"**。

#### 什么是 Vibe Coding？

Vibe Coding 指的是一种全新的编程方式——开发者不再逐行手写代码，而是用自然语言描述自己想要什么，让 AI（如 Claude、GPT、Cursor 等）来生成代码。整个过程更像是一种"感觉流"（vibe）：你和 AI 对话，描述意图，AI 产出代码，你运行、测试、反馈，再继续迭代。

> Karpathy 的原话大意："我完全沉浸在 vibe 里，拥抱 LLM，忘记代码的存在。我只是描述我想要什么，AI 帮我做出来。"

#### 核心特征

1. **意图驱动**：你说"我要一个带搜索功能的资源导航页"，而不是思考用 React 还是 Vue、怎么布局
2. **AI 生成代码**：大模型负责将自然语言意图翻译成可运行的代码
3. **快速迭代**：不满意就继续对话修改，像聊天一样调代码
4. **降低门槛**：非程序员也能"写"出能用的软件

#### Vibe Coding vs 传统编程

| 维度 | 传统编程 | Vibe Coding |
|------|----------|-------------|
| 输入 | 精确的代码语法 | 自然语言描述 |
| 思维模式 | 先想清楚怎么实现 | 先描述想要什么效果 |
| 迭代速度 | 改代码 → 编译 → 调试 | 对话 → 生成 → 运行 |
| 适用场景 | 生产级系统、底层逻辑 | 原型验证、工具脚本、个人项目 |
| 可控性 | 完全掌控每一行 | 需要审查 AI 输出 |

#### 优势

- **效率极高**：原型开发从几天缩短到几小时甚至几分钟
- **降低认知负荷**：不需要记住所有 API 和语法细节
- **探索性强**：快速试错，想到什么试什么
- **赋能非技术人员**：产品经理、设计师也能直接"写"出可用的 demo

#### 风险与挑战

- **代码质量参差不齐**：AI 可能生成有 bug、不安全或不可维护的代码
- **"黑盒"问题**：如果开发者不理解生成的代码，出了问题难以排查
- **不适合复杂系统**：大型项目的架构设计、性能优化仍需专业判断
- **安全风险**：AI 可能引入漏洞或不安全的依赖

#### 最佳实践

1. **理解生成的代码**：不要盲目信任 AI 输出，至少要能读懂
2. **小步迭代**：一次只让 AI 做一件事，逐步构建
3. **版本控制**：用 Git 追踪每一次 AI 生成的变化
4. **AI 辅助 + 人工审查**：AI 写代码，人做 Code Review
5. **从简单场景开始**：先用 Vibe Coding 做工具脚本、原型，积累经验后再用于复杂场景

## 和obsidian联动

### obsidian介绍

**Obsidian** 是一款基于本地 Markdown 文件的**知识管理软件**，由 Shida Li 和 Erica Xu 于 2020 年发布。它被称为"第二大脑"（Second Brain）工具，核心理念是**你的数据永远属于你**——所有笔记以纯文本 Markdown（`.md`）格式存储在本地，不依赖任何云端服务。

#### 核心特性

| 特性 | 说明 |
|------|------|
| **本地优先** | 所有笔记存储在本地文件夹，完全离线可用，无需联网 |
| **双向链接** | 通过 `[[笔记名]]` 语法建立笔记之间的网状关联，而非传统的文件夹层级 |
| **关系图谱** | 可视化展示所有笔记之间的引用关系，一眼看清知识结构 |
| **插件生态** | 超过 2000 个社区插件，覆盖日历、看板、白板、数据库查询等各种场景 |
| **Markdown 原生** | 纯文本格式，可用任何编辑器打开，数据永不过时 |
| **跨平台** | Windows、macOS、Linux、iOS、Android 全覆盖，通过 iCloud/Git/Syncthing 等同步 |

#### 为什么是"第二大脑"？

传统笔记软件（如 Evernote、Notion）是"收集箱"模式——你往里面存东西，用的时候搜索。Obsidian 的哲学不同：

- **网状思维**：人脑不是文件夹层级结构，而是通过联想连接的网状结构。双向链接正是模拟这种思维模式。
- **涌现结构**：不需要预先设计分类体系，写多了自然形成知识网络
- **渐进式整理**：可以先随便写，等笔记多了再通过链接、标签、MOC（内容地图）逐步结构化

#### 与 AI Agent 的结合

Obsidian 与 AI Agent 是天作之合：

1. **本地文件 = 完美上下文**：Agent 可以直接读写 `.md` 文件，零门槛接入你的知识库
2. **结构化检索**：通过 Dataview 插件，Agent 可以像查数据库一样查询你的笔记
3. **知识图谱增强**：Agent 可以分析你的笔记网络，发现关联、推荐链接、补全知识盲区
4. **自动化工作流**：结合 Templater、QuickAdd 等插件，Agent 可以自动生成日记、周报、项目总结
5. **与 Zotero 联动**：文献笔记 + 个人思考在同一个 Obsidian 知识库中融合，Agent 可以同时理解"别人的观点"和"你的想法"

> **一句话总结**：Obsidian 是你的"离线第二大脑"，AI Agent 是你的"在线思考助手"。两者结合，你拥有一个既私密安全、又智能强大的知识管理系统。
